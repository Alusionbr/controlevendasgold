# Auditoria QA e segurança — 2026-07-12

## Escopo

Auditoria do Controle360 publicado no GitHub Pages e do projeto Supabase
`zcwnfrhtlhjfprsjktlx`, cobrindo autenticação, RLS, isolamento entre vendedores,
fluxos administrativos, integridade financeira, estoque, carrinho público,

## Testes aprovados

- Login administrativo e criação de vendedores pela Edge Function.
- Isolamento de clientes entre dois vendedores distintos.
- Bloqueio de tentativa de elevação de vendedor para administrador.
- Bloqueio de criação de vendedores por outro vendedor.
- Ocultação de fornecedores, compras, estoque central e custos para vendedor.
- Bloqueio real de contas com `profiles.active = false`, inclusive via API.
- Carrinho público: consulta, primeiro envio e bloqueio de reenvio com HTTP 409.
- Compra, atualização de estoque e cálculo de custo médio ponderado.
- Ficha técnica, consumo de insumo, produção e custo unitário.
- Venda, receita líquida, taxas, CMV, lucro e baixa de estoque.
- Consignação, pedido e tarefa administrativa.
- Regras de preço sugerido e preço mínimo do vendedor.
- Sintaxe de todos os módulos JavaScript e disponibilidade de todos os assets publicados.
- Auditoria de dados existentes sem estoques negativos ou inconsistências matemáticas.

## Correções aplicadas

1. Carrinho público passou a aceitar somente um envio e usa atualização condicional
   para bloquear requisições concorrentes. Upload órfão é removido em caso de corrida.
2. Funções `SECURITY DEFINER` internas deixaram de ser executáveis por anônimos.
3. Policies antigas foram restringidas de `PUBLIC` para `authenticated`.
4. Conta desativada passou a ser bloqueada nas policies, mantendo somente a leitura
   do próprio perfil para a mensagem de desativação.
5. A view mascarada de produtos agora exige explicitamente perfil ativo no mesmo negócio.
6. Foram adicionadas restrições de integridade para estoque, compras, produção,
   pedidos, consignações, preços e quantidades aprovadas.
7. Todas as chaves estrangeiras apontadas pelo advisor receberam índice de cobertura.

## Resultado dos advisors

- Chaves estrangeiras sem índice: **0**.
- Policies permissivas sobrepostas: reduzidas de **153** para **37**.
- Permanecem avisos de otimização de `auth.uid()` e índices recém-criados ainda sem uso,
  esperado enquanto a base é pequena.
- A view `seller_products` permanece `security_definer` por arquitetura: ela mascara
  custos e estoque central sem conceder leitura direta de `products` ao vendedor.
  A definição agora valida identidade, perfil ativo e negócio explicitamente.

## Riscos residuais e próximos passos

- Ativar proteção contra senhas vazadas no painel Supabase Auth.
- Os fluxos de compra, produção e venda ainda fazem várias gravações sequenciais no
  navegador. Uma falha de rede no meio pode deixar operação parcial. Recomenda-se
  migrar esses fluxos para RPCs transacionais no banco.
- GitHub Pages não envia Content-Security-Policy. Avaliar CSP via meta tag ou hospedagem
  que permita cabeçalhos personalizados.
- Contas QA de Auth criadas durante os testes ficaram com perfil inativo e sem acesso.
  Podem ser excluídas posteriormente no painel Auth para reduzir ruído.
- Repetir testes visuais manuais em desktop e celular antes de uma liberação comercial,
  especialmente upload de comprovante, navegação inferior e importação/exportação Excel.
